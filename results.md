# パッチール模様 全数探索 — 結果報告

実施日: 2026-06-11
環境: Windows 11 / Ryzen 7 9700X (8C16T) / Rust 1.96 (x86_64-pc-windows-gnu)

---

## 結論

第三世代（ルビー・サファイア・エメラルド・FRLG）のパッチールが取りうる
「見た目として区別できる模様」の厳密な数は：

```
3,880,873,934 通り
```

| 量 | 値 | 備考 |
|---|---|---|
| 性格値の総数 (2³²) | 4,294,967,296 | 理論上の最大 |
| Bulbapedia の概算上界 | 3,945,136,128 | 「完全はみ出しのみ同一視」の位置数の積 |
| 本研究の per-spot 積上界 | 3,913,023,488 | 253×256×236×256（見た目ベース・独立仮定） |
| **厳密ユニーク数** | **3,880,873,934** | 全 2³² 実描画・重複除去 |
| PID→見た目の縮退総数 | 414,093,362 | 2³² − 厳密値 |
| スポット間相互作用による縮退 | 32,149,554 | per-spot 積 − 厳密値（重なり・共食いの寄与）|
| Bulbapedia 上界との差分 | 64,262,194 | |

- 厳密値は理論最大の **90.36%**。
- 「2匹のパッチールが同じ見た目になる確率は40億分の1未満」という公式図鑑の
  記述に対し、正確には **約 38.8 億通り** が正解。

### per-spot 単独の異なり数（見た目ベース）

| スポット | 基準座標 | 異なり数 | Bulbapedia (位置ベース) |
|---|---|---|---|
| 0 (左耳〜顔左上) | (16, 7) | 253 | 254 |
| 1 (右耳) | (40, 8) | 256 | 256 |
| 2 (顔左下) | (22, 25) | 236 | 237 |
| 3 (顔右下) | (34, 26) | 256 | 256 |

Bulbapedia の数字は「スポットが完全にスプライト外（肌色外）に出る配置をすべて
同一視」した*位置*の数。部分クリップ同士で見た目が一致するケース（スポット0と
2に各1組存在）を数え漏らしているため、各スポットでちょうど 1 だけ大きい。
本研究の値は最終描画ピクセルの一致で判定したもの。

## 「見た目が同じ」の定義と移植の根拠

pokeemerald `src/pokemon.c` の `DRAW_SPINDA_SPOTS` マクロを忠実に移植した。
原典から確定した仕様：

1. **肌色判定**: スポットのピクセルは、下地のパレットインデックスが 1〜3
   （明るい体色）のときだけ +4 され、5〜7（模様色）になる。
   目・耳の渦・輪郭・背景には描かれない（クリップ縮退の源）。
2. **重なりの挙動**: 一度 5〜7 になったピクセルは範囲外なので、後続のスポットは
   スキップする。したがって**最終画像 = ベース + 4×(4スポット被覆の和集合 ∩ 肌色領域)**
   と数学的に等価で、描画順は結果に影響しない（重なり縮退の源）。
3. **座標の u8 巻き戻り**: スポット0（基準 y=7）の y ニブルが 0 のとき
   y = −1 → 255 となり、最初の行は可視フレーム（64×64）外のバッファ
   （オフセット 7936..8191、ゲーム内の 0x2000 スロット内）に書かれ、
   見た目には現れない。
4. **対象スプライト**: 戦闘前面絵 = `anim_front.png` 第1フレーム
   （`gMonFrontPicTable` 経由）。エメラルドのパッチールは模様アニメを避ける
   ため前面1フレームのみ。第2フレームは全ゼロ。

### 第三世代3作品で値が同一である根拠

- **FRLG**: 前面絵・スポット基準座標ともエメラルドと完全一致（pokefirered で確認）。
- **ルビサファ**: 前面絵はエメラルドを 7px 下にシフトしたもの、スポット基準座標も
  全て y+7（(16,14),(40,15),(22,32),(34,33)）。体に対する模様の相対配置は同一。
  エメラルドで y=255 に巻き戻って消える行は、RS では「体の上方の背景
  （肌色でないので元々描かれない）」に対応し、見た目は変わらない。
  描画コード（pokeruby `pokemon_3.c` の `DrawSpindaSpots`）も意味的に同一。

## 検証チェーン（数字を信じる理由）

1. **独立2実装の一致**: Python 直訳版（4bpp タイルバッファ上でマクロを直訳）と
   Rust 忠実移植版が、決定的に選んだ 256 PID で全ピクセル一致。
2. **忠実版 vs 高速版**: 本番用ビットマスク実装（和集合∩肌色）の出力から
   フル画像を再構成し、忠実版とランダム **1億 PID** でピクセル完全一致。
3. **ROM 実バイトとの照合**: スポット形状はpokeruby の `spot_*.bin`
   （ROM 由来の生バイナリ）と全 64 行一致。
   ※ここで重大バグを検出：gbagfx の 1bpp 変換は「u16 の bit15 = PNG の左端」
   なのに対し、ゲームは bit0 から描くため、**ゲーム内の模様は PNG の左右反転**。
   PNG を素直に読むと鏡像の模様で全数探索することになる。
4. **先行研究との整合**: per-spot 異なり数が Bulbapedia の位置ベース値
   {256,256,254,237} と「各スポットで高々1小さい」という理論どおりの関係。
5. **HLL 近似との整合**: HyperLogLog (p=14, 標準誤差0.81%) の推定
   3,896,771,644 に対し、厳密値は 0.5σ 以内。
6. **収集件数の検算**: 16 パスの収集件数合計 = 4,294,967,296 = 2³² と完全一致
   （全 PID がちょうど1回ずつ数えられた）。
7. **別ハッシュ・別分割数での再計算**: xxh3-128 のシードを変えた別ハッシュ族
   ×32 パスで全数を再実行し、同一の厳密値を得た（下記）。

ハッシュ衝突について: 128bit ハッシュで 4.3×10⁹ 件の場合、衝突が1組でも
起きる確率は約 3×10⁻²⁰ で無視できる。さらに 7. の独立ハッシュでの一致により
「両方のハッシュで同時に衝突が起きた」場合を除き厳密性が保証される。

### 検算結果（シード 1234567890123、32 パス）

独立したハッシュ族・異なる分割数で全 2³² を再実行した結果：

```
厳密ユニーク数: 3,880,873,934 （本計算と完全一致）
収集件数合計:   4,294,967,296 = 2³² （全 PID がちょうど1回）
総経過時間:     500.4 秒
```

## 性能

| フェーズ | 時間 |
|---|---|
| HLL 近似（全 2³² 描画+推定） | 10.6 秒 |
| 厳密カウント（16 パス、計 16×2³² 描画） | 258.4 秒 |

1 PID あたりの描画+ハッシュは実測 ~40ns（シングルスレッド換算、16スレッド並列）。
高速化の要点: 模様描画を「事前計算した配置マスク（肌色 AND 済み u64×16 行）の OR」
に還元し、PID 上位 24bit（スポット1〜3）を外側ループで固定して部分マスクを再利用。

## 再現手順

```powershell
# 1. アセット抽出（pokeemerald / pokeruby / pokefirered をスパースクローン済みであること）
python tools\extract_assets.py   # data/ に base.bin, palette.bin, spots.bin を生成・ROM照合

# 2. テストベクタ生成 + サンプル PNG（目視確認）
python tools\proto_render.py     # data/test_vectors.bin, out/*.png

# 3. Rust 検証と本番
cd rust\spinda-count
cargo build --release
.\target\release\spinda-count.exe verify 100000000   # 実装一致検証
.\target\release\spinda-count.exe perspot            # per-spot 異なり数
.\target\release\spinda-count.exe hll                # 近似カウント
.\target\release\spinda-count.exe exact 16           # 厳密カウント
.\target\release\spinda-count.exe exact 32 1234567890123  # 別ハッシュで検算
```

注意: pokeemerald/pokeruby/pokefirered のアセットは解析・検証目的でのみ参照。
ROM 由来データ（data/ 以下）は再配布しない（.gitignore 済み）。

  本結果は第三世代（GBA）のみ。
