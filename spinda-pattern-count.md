# パッチール模様 全数探索プロジェクト

第三世代（GBA）のパッチールが取りうる「見た目として区別できる模様」の **正確な数** を、全 2³² 通りの性格値を総当たりで実描画し、重複を除いて数え上げて確定させるプロジェクト。

---

## 1. 背景と目的

### 何が分かっていて、何が未解決か

パッチールの模様は性格値（32ビット）で決まる。4か所の模様それぞれに8ビットが割り当てられ、下位4ビットが x 座標オフセット、上位4ビットが y 座標オフセットを司る。

- bit0–7 → 左上の模様
- bit8–15 → 右上の模様
- bit16–23 → 左下の模様
- bit24–31 → 右下の模様

（※これは第三世代＝リトルエンディアン読みの順。BDSP のみビッグエンディアンで逆順になるが、本プロジェクトの対象外）

したがって理論上の組み合わせは **2³² = 4,294,967,296 通り**。ただし「性格値の数」と「見た目の数」は一致しない。理由は2つ：

1. **クリップ** — 模様のスタンプが顔の描画領域からはみ出すと、別の座標でも見た目が同じになる
2. **重なり（eclipse）** — 4つの模様同士が重なると、別配置でも同じ見た目になる

### 先行研究の到達点と「隙間」

Bulbapedia には既存の概算がある：各模様が単独で取りうる異なり数を実測し、`256 × 256 × 254 × 237 = 3,945,136,128` とする計算。

**ただしこれは「4つの模様は互いに干渉しない」という独立性を仮定した掛け算**で、クリップ（模様1個のはみ出し）は織り込めるが、**模様どうしの重なりによる縮退は数え切れていない**。Bulbapedia 自身が「左上の模様が左下を覆い隠す場合などがあるため、この合計でもわずかに多すぎる」と明記し、確定値は出していない。

→ **本プロジェクトのゴール**：全 2³² を実際に描画してピクセル単位で照合し、重なりも含めた縮退をすべて潰した **厳密なユニーク数** を確定する。期待される結果は `3,945,136,128` よりわずかに小さい値で、その差分が「重なり由来の縮退の数」になる。これは（第三世代でも）まだ公開されていない数字。

### 成果物

- パッチール模様の厳密なユニーク数（整数1個）
- `3,945,136,128 − 厳密値` ＝ 重なり由来の縮退数
- 検証可能な再現コードと、結果を裏付けるログ

## 2. 正しさの基準（ここが最重要）

「見た目が同じ」の定義は、**最終的に描画されたスプライト画像のピクセルが一致すること**。これを揺るがせないために、模様の描き込みは **pokeemerald のオリジナル処理を忠実に移植する**ことを唯一の正解とする。自前で「だいたいこう描くはず」と推測してはいけない。

参照すべき pokeemerald の実体：

- `DrawSpindaSpots`（模様を性格値からスタンプする関数）
- `gSpindaSpotGraphics`（4模様それぞれの基準座標と 16×16 のスポット形状データ）
- パッチールのベース前面スプライト（インデックスカラー）とパレット

リポジトリ: https://github.com/pret/pokeemerald

**特に注意すべき仕様（移植時に実コードで必ず確認すること）**：

- スポットは単純な矩形コピーではなく、**下地が特定の肌色パレットインデックスのピクセルだけ**を模様色に変える方式。だから模様が顔の外（背景・目・耳の渦など肌色でない領域）にかかる部分は描画されず、これがクリップ縮退の源になる。
- 4模様は順番に描かれる。**重なった領域で後段の模様がどう作用するか**（既に模様色になったピクセルを肌色判定でスキップするのか、インデックス加算で別色になるのか）が重なり縮退の挙動を決める。**ここを推測で実装すると答えがズレる**ので、必ず原典の挙動を再現する。

この「最終描画ピクセルで判定する」方針なら、クリップも重なりも特別扱い不要で、すべて自然に縮退として数え上げられる。

## 3. アーキテクチャ

### 全体フロー

```
ベーススプライト＋パレット＋スポットデータ（pokeemerald 由来）をロード
        │
        ▼
PID = 0 .. 2³²-1 を総当たり（並列）
        │
        ├─ DrawSpindaSpots を忠実適用して領域を描画
        ├─ 模様可動範囲だけを切り出してキー化（後述）
        └─ キーをハッシュ
        │
        ▼
全ハッシュの異なり数を数える（重複除去）
        │
        ▼
厳密ユニーク数を出力 ＋ 3,945,136,128 との差分を報告
```

### キーの作り方（描画範囲の限定）

全スプライトをハッシュするのは無駄。`gSpindaSpotGraphics` の基準座標＋オフセット範囲 `[0,15]`＋スポット寸法 `16×16` から、**4スポットが触れうるピクセルの外接矩形（bounding box）** を事前計算し、その小矩形だけを切り出す。

ベーススプライトは不変なので、各 PID で変わるのは「どのピクセルがどの模様色に変わったか」だけ。よって **小矩形の最終インデックス列（または『下地と異なるピクセルの位置と色』のビットマスク）をキー** にすれば、見た目の同一性を過不足なく表現できる。

### 言語・ツール

- **本番（全数探索）は Rust か C/C++ 必須**。4.3億回の描画＋ハッシュは Python では遅すぎる。Rust 推奨（`rayon` で並列化、`twox-hash`/`xxhash-rust` 等で高速ハッシュ、Claude Code でも書きやすい）。
- **プロトタイプ・検証・可視化は Python で可**（1個や数千個の描画で十分なフェーズ）。
- **並列化**: Ryzen 7 9700X は8コア16スレッド。`rayon` でレンジ並列にすれば描画フェーズは数分規模で終わる見込み。
- GPU（RTX 5060 Ti）は使わなくても十分間に合う。小さなマスク描画は CPU SIMD で足りる。使うなら重複除去フェーズの大規模ソートを CUDA に載せる選択肢はあるが、初手では不要。

## 4. 重複除去（ボトルネックはここ）

4.3億個のキーの異なり数を数える。64bit ハッシュなら全部で約34GiB——多くの環境で RAM に乗り切らないため、戦略を選ぶ。

### 段階的に3つ用意する

**(A) 近似カウント（最初のマイルストーン）**
HyperLogLog で異なり数を推定。メモリ数KB、描画フェーズとほぼ同じ時間（数分）で「だいたい何種類か」が出る。まず全体像をつかみ、`3,945,136,128` 近傍に着地するかを確認する用。

**(B) 外部基数分割による厳密カウント（本命）**
ハッシュの上位ビットでキーを N 個（例: 256）のバケットにディスク分割し、各バケットを RAM に載せてハッシュセットで重複除去。1バケットあたり ~1670万キー程度なので RAM に収まる。全体で数十分〜1時間規模。これで厳密値を出す。

**(C) 構造を利用した検算（任意・発展）**
模様が「互いに重ならず、かつクリップもされない」PID の集合では、見た目数は各模様の独立な異なり数の積に分解できる（＝先行研究の掛け算が正しく効く領域）。重なり/クリップが起きる PID 部分集合だけを抜き出して個別に潰せば、(B) と独立な経路で同じ答えに到達できるはず。(B) の検算として使う。

### ハッシュ衝突への対策

64bit ハッシュを 4.3億件に使うと衝突期待値が無視できない水準になる。「厳密」を名乗るなら **128bit ハッシュを使う**か、64bit で数えたうえで衝突候補を実キーで突き合わせて補正する。最初は 128bit で安全側に倒すのが楽。

## 5. 検証計画（数字を信じる前にやる）

全数探索の結果は、**移植が忠実である**ことが担保されて初めて意味を持つ。順序：

1. **単体描画の一致確認** — 既知の性格値をいくつか選び、自作描画の出力を既存の Spinda ジェネレーター（pokeos / web Spinda 等）やゲーム内スクショと突き合わせる。ピクセル単位で一致するまで本番に進まない。
2. **境界ケース** — 模様が完全にはみ出す PID、明らかに重なる PID を手で作り、クリップ・重なりの挙動が原典どおりか確認。
3. **先行研究との整合** — まず近似カウント (A) が `3,945,136,128` 付近かを確認。大きく外れたら移植かキー設計のバグ。
4. **厳密値の健全性** — (B) の厳密値が `3,945,136,128` 以下で、かつ妥当に近い差分に収まるか。(C) と一致するか。

## 6. 実装フェーズ

### Phase 0: データ取得と土台
- pokeemerald をクローンし、`gSpindaSpotGraphics`・`DrawSpindaSpots`・ベーススプライト・パレットを特定して抽出
- ベーススプライトをインデックスカラーのまま扱える形式に変換
- スポット可動範囲の bounding box を算出

### Phase 1: 忠実な描画器（1個）
- `render(pid) -> 小矩形のインデックス列` を実装
- pokeemerald の肌色判定・描画順・重なり挙動を忠実に再現
- Phase 5 の検証1・2を通す

### Phase 2: キー化とハッシュ
- 小矩形 → キー（下地と異なるピクセルの位置＋色のビットマスク等）→ 128bit ハッシュ
- 同一見た目が同一キーになることを少数例で確認

### Phase 3: 近似全数探索（マイルストーンA）
- `rayon` で PID 全域を並列描画＋HyperLogLog
- 推定ユニーク数を出力、`3,945,136,128` 近傍かを確認

### Phase 4: 厳密全数探索（本命B）
- ハッシュを外部基数分割 → バケットごとに RAM 内重複除去
- 厳密ユニーク数を確定、衝突補正を実施

### Phase 5: 検算と報告（発展C）
- 構造分解 (C) で独立に再計算し (B) と照合
- 最終レポート: 厳密値、`3,945,136,128` との差分（＝重なり縮退数）、再現手順

---

## 7. 性能の見積もり（目安）

- **描画＋ハッシュ**: 1 PID あたり数百ns想定。4.3億件を16スレッドで回すと描画フェーズ自体は **数分** 規模。
- **重複除去**: 近似(A)なら描画と同程度（数分）。厳密(B)は外部分割の I/O が効いて **数十分〜1時間**。
- 全体として「厳密値まで1時間以内」が現実的な着地点。まず (A) で数分で全体像 → (B) で時間をかけて確定、の二段構えが効率的。

---

## 8. スコープ外・注意

- **対象は第三世代のみ**（RSE / FRLG / Emerald。模様描画ロジックは第三世代内で共通なので「第三世代の値」として1つに定まる）。Switch 世代（BDSP 等）のデータマイニングは行わない。
- BDSP はベーステクスチャの解像度・形状が異なり縮退の起き方も変わるため、仮にやるなら別プロジェクト・別の数字になる。今回は扱わない。
- pokeemerald のアセット・コードはあくまで解析/検証目的で参照する。成果物として ROM 由来データそのものを再配布しないこと。

## 9. 最初の一手（Claude Code への指示の出し方）

1. 「pokeemerald をクローンして `DrawSpindaSpots` と `gSpindaSpotGraphics` を見つけ、模様描画の正確な仕様（肌色判定・描画順・重なり挙動）を要約して」
2. 「その仕様に忠実な単体描画器を Rust で実装し、既知 PID 数例の出力を検証して」
3. 「検証が通ったら HyperLogLog で近似全数探索して、`3,945,136,128` 近傍か確認して」
4. 「最後に外部基数分割で厳密ユニーク数を確定して」

Phase 1 の忠実移植が全体の成否を決める。ここだけは推測で進めず、必ず原典コードに当たること。
